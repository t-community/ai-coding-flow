AI Coding Flow

An autonomous AI coding workflow that watches GitHub and GitLab issues, writes code to resolve them, runs tests, opens a pull request, and posts a code review — all without human intervention.

FastAPI Aider · OpenCode GitHub · GitLab OpenAI-compatible LLM

What it does

You assign an issue, AI Coding Flow picks it up, writes code, and opens a PR — you only decide whether to merge.

Key features

🤖

Pluggable engines

Switch between Aider and OpenCode per issue using labels.

🔁

Retry loop

Runs tests after every coding attempt and feeds failures back to the LLM.

🔍

AI code review

A fresh LLM session reviews the diff and posts a structured comment.

🏷️

Label status

Issue labels track progress: processing → done / failed / needs clarification.

🌐

GitHub + GitLab

Supports both platforms, including self-hosted GitLab.

📴

Fully offline

Any OpenAI-compatible endpoint works — local Ollama, OpenRouter, etc.

Non-goals

  • Auto-merging PRs/MRs — the human always clicks merge
  • Parallel issue processing — jobs run sequentially
  • Handling issue comments or updates — only "issue opened" and label events trigger work

Quick Start

Get AI Coding Flow running in under 5 minutes.

Prerequisites

  • Python 3.11+
  • aider-chat installed and on $PATH
  • A GitHub or GitLab account with a repository
  • An OpenAI-compatible LLM endpoint (local Ollama, OpenRouter, etc.)
  • A publicly reachable URL for your server (or a tunnel like ngrok)

Installation

  1. Clone the repository 
    git clone https://github.com/t-community/ai-coding-flow
cd ai-coding-flow
  2. Install Python dependencies 
    pip install -r requirements.txt
  3. Create your .env file 
    cp .env.example .env
# then edit .env with your values
  4. Start the server 
    uvicorn server:app --host 0.0.0.0 --port 8000

    Or with Docker:

    docker build -t ai-coding-flow .
docker run --env-file .env -p 8000:8000 ai-coding-flow
  5. Configure webhooks on your platform

    See Webhook Setup for step-by-step instructions.

  6. Open a test issue

    Create an issue in your repository. Within seconds the ai: processing label appears, and minutes later a PR is created with a review comment.

Tip: Set AIDER_VERBOSE=true in your .env to see the full Aider output in the server logs — useful for debugging why an AI run produced no changes.

Verifying the server

curl http://localhost:8000/health
# → {"status":"ok"}

The interactive API docs are available at /docs (Swagger UI) and /redoc (ReDoc).

Webhook Setup

AI Coding Flow needs to receive webhook events from your Git platform whenever an issue is opened or an agent: label is added.

Public URL required: GitHub and GitLab must reach your server. Use ngrok http 8000 during development, or deploy to a server with a public IP.

GitHub

  1. Go to Settings → Webhooks → Add webhook

    Navigate to your repository on GitHub.

  2. Fill in the form
    Payload URLhttps://your-server.example.com/webhook/github
    Content typeapplication/json
    SecretSame value as WEBHOOK_SECRET in your .env
    EventsSelect Issues only
  3. Click "Add webhook"

    GitHub will send a ping event. Check your server logs for a 200 response.

Events triggered: The Issues event covers both opened (new issue) and labeled (label added) — both are needed for AI Coding Flow.

GitLab

  1. Go to Settings → Webhooks

    Navigate to your project in GitLab.

  2. Fill in the form
    URLhttps://your-server.example.com/webhook/gitlab
    Secret tokenSame value as WEBHOOK_SECRET
    TriggerEnable Issues events
    SSL verificationEnable if your server has a valid TLS certificate
  3. Click "Add webhook" and test it

    GitLab provides a "Test" button — send an Issues event and verify the server returns 200.

Security

GitHub webhooks are verified with HMAC-SHA256. GitLab webhooks use a shared secret token. Both are compared with constant-time comparison to prevent timing attacks. Requests with invalid signatures return HTTP 403.

Labels

AI Coding Flow uses two label namespaces: ai: for status tracking and agent: for engine selection.

Status labels (managed automatically)

LabelSet when
ai: processing Job is picked up by the worker — coding is in progress
ai: done PR created and review comment posted successfully
ai: failed Engine exhausted retries or an unexpected error occurred
ai: needs clarification Engine ran but made zero code changes — the issue needs more detail

Engine labels (set by you)

LabelEffect
agent: aider Use the Aider engine for this issue
agent: opencode Use the OpenCode engine for this issue

Re-triggering: Adding an agent: label to an issue fires a webhook that re-enqueues the job. If the previous run failed with Aider, add agent: opencode to retry with a different engine.

Label selection logic

  1. Scans the issue's current labels for the first one starting with agent:
  2. If found, uses that engine by name (falls back to Aider if the name is unrecognized)
  3. If no agent: label is present, uses the DEFAULT_AGENT setting

Multiple agent: labels: If an issue has both agent: aider and agent: opencode, whichever appears first in the API response is used. Remove the unwanted one to be explicit.

Request lifecycle

  1. Webhook received

    server.py verifies the signature and enqueues the job as a background task, returning 200 within milliseconds.

  2. Engine selection

    worker.py reads the issue's labels and calls _pick_engine() to select Aider, OpenCode, or the configured default.

  3. Repository preparation

    agent.py clones to /tmp/ai-coding-flow/<issue_number>/ (or fetches if cached) and creates branch ai/issue-{n}-{slug}.

  4. AI coding loop

    The selected engine is called with the prompt. If TEST_CMD is set, tests run after each attempt; failures feed back to the engine. Repeats up to MAX_RETRIES times.

  5. PR creation and review

    The branch is pushed, a PR/MR is created via the platform API, then a separate AI session reviews the diff and posts a comment.

Concurrency model

FastAPI handles concurrent HTTP requests asynchronously. AI coding work runs in a thread pool via asyncio.to_thread(). The job queue is sequential — one issue processes at a time — to avoid git conflicts.

Engine System

Engines are pluggable AI coding backends. Each engine receives a repository path, a prompt, and the settings object, and is responsible for making code changes to the working tree.

Built-in engines

Aider (default)

Labelagent: aider
Binaryaider (must be on $PATH)
LLM env varOPENAI_API_BASE, OPENAI_API_KEY
Flags--yes --auto-commits --no-stream --message <prompt>
Verbose loggingSet AIDER_VERBOSE=true

OpenCode

Labelagent: opencode
Binaryopencode (must be on $PATH)
LLM env varOPENAI_BASE_URL, OPENAI_API_KEY
Flags--model <model> --message <prompt>

Note: OpenCode uses OPENAI_BASE_URL while Aider uses OPENAI_API_BASE. Both are populated from the same OPENAI_API_BASE config value.

AgentEngine ABC

engines/base.py
from abc import ABC, abstractmethod
from pathlib import Path
from config import Settings

class AgentEngine(ABC):
    @property
    @abstractmethod
    def name(self) -> str: ...

    @abstractmethod
    def run(self, repo_path: Path, prompt: str, settings: Settings) -> str:
        """Returns combined stdout+stderr. Exceptions propagate."""
        ...

Platforms

The platform abstraction provides a uniform interface to GitHub and GitLab. All API interactions go through the GitPlatform ABC.

GitPlatform interface

platforms/base.py
class GitPlatform(ABC):
    def get_issue(self, number: int) -> Issue: ...
    def create_pr(self, branch: str, title: str, body: str) -> str: ...
    def post_comment(self, issue_number: int, body: str) -> None: ...
    def set_label(self, issue_number: int, label: str) -> None: ...
    def remove_label(self, issue_number: int, label: str) -> None: ...
    def get_labels(self, issue_number: int) -> list[str]: ...
    # Returns [] if issue not found (never raises)

Extending

Adding a new AI engine

  1. Create engines/myengine.py 
    engines/myengine.py
    import subprocess
from pathlib import Path
from config import Settings
from engines.base import AgentEngine

class MyEngine(AgentEngine):
    @property
    def name(self) -> str:
        return "myengine"

    def run(self, repo_path: Path, prompt: str, settings: Settings) -> str:
        result = subprocess.run(
            ["myengine-cli", "--prompt", prompt],
            cwd=str(repo_path), capture_output=True,
            text=True, timeout=600,
        )
        return (result.stdout + result.stderr).strip()
  2. Register in engines/__init__.py 
    _ENGINES = {
    "aider":    AiderEngine,
    "opencode": OpenCodeEngine,
    "myengine": MyEngine,   # ← add this
}
  3. Create the label in your repository

    Add an agent: myengine label to your GitHub/GitLab project, then apply it to an issue.

Adding a new platform

  1. Implement platforms/bitbucket.py

    Subclass GitPlatform and implement all 6 abstract methods.

  2. Register in platforms/__init__.py
    if settings.platform == "bitbucket":
    return BitbucketPlatform(...)
  3. Add a webhook endpoint in server.py

    Add @app.post("/webhook/bitbucket") with appropriate verification.

  4. Update config.py

    Add the new platform to the Literal type and any required token fields.

Testing

Running the test suite

# All tests
pytest -v

# Specific module
pytest tests/test_engines.py -v

# Filter by keyword
pytest -k "get_labels" -v

All tests are unit tests — no network calls, no LLM invocations. External dependencies are mocked with unittest.mock.
